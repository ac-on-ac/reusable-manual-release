# reusable-manual-release

A reusable GitHub Actions workflow that manually creates a versioned GitHub release with AI-generated release notes summarised from the git history since the last release.

## Purpose

This workflow:

1. Computes the next semantic version based on the latest tags and the chosen bump type
2. Generates release notes by summarising `git log` and `git diff --stat` output using the GitHub Models API (`gpt-4o-mini`)
3. Creates a GitHub release (and tag) at `HEAD` with those notes

It supports both stable releases and release candidates, and correctly increments RC numbers when successive release candidates are cut for the same target version.

## Inputs

| Input | Required | Type | Description |
|---|---|---|---|
| `version_type` | Yes | `string` | One of `patch`, `minor`, `major`, `rc-patch`, `rc-minor`, `rc-major` |

## Permissions

The calling workflow must grant these permissions:

| Permission | Reason |
|---|---|
| `contents: write` | Create the release and tag |
| `models: read` | Call the GitHub Models API to generate release notes |

## Calling this workflow

Add a workflow file to your repository:

```yaml
name: Create Release

on:
  workflow_dispatch:
    inputs:
      version_type:
        description: 'Version bump type'
        required: true
        type: choice
        options:
          - patch
          - minor
          - major
          - rc-patch
          - rc-minor
          - rc-major

permissions:
  contents: write
  models: read

jobs:
  release:
    uses: ac-on-ac/reusable-manual-release/.github/workflows/release.yml@v1.0.0
    with:
      version_type: ${{ inputs.version_type }}
```

Then trigger it from **Actions → Create Release → Run workflow** in your repository's GitHub UI.

## Versioning behaviour

Versions are computed from your repository's git tags. The workflow maintains two references internally:

- **Latest tag** (any, including RCs) — used as the base for `git diff` so release notes only cover what changed since the last run
- **Last stable tag** (non-RC) — used as the semver base so version numbers always advance from a stable baseline

### Stable releases

| Last stable tag | Input | Result |
|---|---|---|
| `v1.2.3` | `patch` | `v1.2.4` |
| `v1.2.3` | `minor` | `v1.3.0` |
| `v1.2.3` | `major` | `v2.0.0` |

### Release candidates

RC numbers increment when you repeatedly choose the same bump type for the same target base. Switching to a different bump type (or a different level) resets the RC counter to `rc.1`.

| Last stable tag | Latest tag | Input | Result |
|---|---|---|---|
| `v1.2.3` | `v1.2.3` | `rc-patch` | `v1.2.4-rc.1` |
| `v1.2.3` | `v1.2.4-rc.1` | `rc-patch` | `v1.2.4-rc.2` |
| `v1.2.3` | `v1.2.4-rc.2` | `rc-patch` | `v1.2.4-rc.3` |
| `v1.2.3` | `v1.2.4-rc.3` | `rc-minor` | `v1.3.0-rc.1` ← base changed, resets |

RC releases are marked as pre-releases on GitHub.

### No prior tags

If the repository has no tags yet, the workflow starts from `v0.0.0` and applies the selected bump, producing `v0.0.1` for `patch`, `v0.1.0` for `minor`, and so on.

## Release notes

Release notes are generated by sending the `git log --oneline` and `git diff --stat` output since the previous tag to `gpt-4o-mini` via the [GitHub Models API](https://docs.github.com/en/github-models). The model produces a short bulleted summary in plain language.

If there are no commits since the last tag, the release notes fall back to a static placeholder.

No additional secrets or tokens are required — the workflow uses the built-in `GITHUB_TOKEN` with the `models: read` permission.
